# 思脑谱 GitHub 工作流

## 1. 开始前同步

- 获取并清理远程引用。
- 切换到 `main`，使用仅快进方式更新本地 `main`。
- 确认本地 `main` 与 `origin/main` 指向同一提交。
- 如果工作区存在无关修改、无法仅快进或出现冲突，停止并向用户说明，不得强行覆盖。

## 2. 创建功能分支

- 正式内容或结构改动不得直接提交到 `main`。
- 从最新 `main` 创建名称清晰的 `codex/<任务说明>` 功能分支。
- 一个分支只承载一个边界明确的任务。

## 3. 修改与验证

- 只修改本次任务需要的文件。
- 按 `.agents/README.md` 和适用的专项 Skill 完成内容核对。
- 检查工作区和差异，确认临时文件、审核记录及无关修改不会进入提交。

## 4. Commit

- 只有用户明确授权 Commit 时才继续。
- 只暂存本次任务明确包含的文件，不默认暂存整个工作区。
- 检查已暂存差异后，使用简短、准确的说明创建 Commit。
- Commit 只保存到本地；尚未 Commit 的修改不会随 Push 上传。

## 5. Push 与 PR

- 只有用户明确授权 Push 时，才把功能分支上传到 `origin`。
- 不得把功能改动直接 Push 到 `main`。
- 只有用户明确要求创建 PR 时，才创建指向 `main` 的 Pull Request。
- PR 应说明改了什么、为什么修改、分类或结构变化，以及验证结果。
- 文章导入默认创建 Draft PR；其他任务根据用户要求决定是否为 Draft。

## 6. Merge

- 未经用户单独明确要求，不得 Merge。
- 用户要求自动合并时，先确认 PR 目标为 `main`、内容范围正确、没有冲突且检查通过。
- 如果仓库没有配置远程检查，明确记录这一事实，并以已经完成的本地验证为依据。
- 确认条件满足后再启用自动合并。

## 7. 合并后的远程清理

- 确认 PR 状态为已合并，并记录合并提交。
- 删除 GitHub 上已经合并的远程功能分支。
- 再次获取并清理远程引用，确认对应的 `origin/<功能分支>` 已消失。

## 8. 合并后的本地收尾

- 切换到本地 `main`，使用仅快进方式更新。
- 确认本地 `main` 与 `origin/main` 指向同一提交。
- 确认合并后的内容在本地 `main` 中完整存在，工作区没有意外修改。
- 最后删除本地功能分支，优先使用安全删除。
- 如果使用 squash merge，只有在 PR 已合并、分支头与 PR 一致、最终内容已经进入 `main` 后，才可处理 Git 的非祖先关系提示。
- 如果任何核对失败，停止清理并向用户说明，不得盲目强制删除。

## 9. 完成交付

向用户报告：

- Commit 和分支；
- PR 地址与状态；
- 验证结果；
- 合并提交；
- GitHub 远程功能分支删除结果；
- 本地 `main` 同步结果；
- 本地功能分支删除结果。
